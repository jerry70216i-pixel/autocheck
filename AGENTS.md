# AGENTS.md

## Project Goal

本專案是工程照片查驗表自動化系統。目標是讓使用者選取或上傳工程照片後，系統能將照片與查驗說明整理到指定表格格式中，減少人工篩選照片與打字整理說明的時間。

## Project Scope

目前準備從 V0 推進到 V1。V1 必須是 V0 的擴充，不是取代 V0。

V0 已完成並必須保留：

- 使用者可以選取或上傳多張照片。
- 系統可以顯示照片預覽。
- 使用者可以手動輸入或修改查驗說明。
- 系統可以依序將照片放入單一固定表格格式。
- 使用者可以匯出 HTML 表格。

V1 只新增：

- 工項資料。
- 查驗說明範本資料。
- 每張照片可選擇工項。
- 選擇工項後可帶入預設查驗說明。
- 使用者仍可手動修改每張照片的查驗說明。
- 匯出表格時可輸出照片與對應說明。
- 資料先使用瀏覽器本機儲存，優先 IndexedDB；若不適合再改 localStorage。

V1 不包含：

- AI 圖像辨識。
- AI 說明生成。
- RAG。
- AI Agent。
- 自動學習。
- 帳號權限。
- 區域網路部署。
- 後端資料庫。
- 多格式輸出。
- 多表格模板。

## Current Version

目前版本準備進入 V1 文件凍結階段。不得直接跳到 V2 到 V5，也不得為未實作版本建立過多抽象、預留大量空架構或新增未使用功能。

若需求不明，先更新 `docs/QUESTIONS.md` 記錄問題與待確認事項，不要自行擴大範圍。

## Architecture Rules

- 任何版本推進不得破壞前一版驗收條件。
- V1 是 V0 的擴充，不是取代。
- 工項與說明不得成為 V0 匯出的必要條件。
- 即使沒有選工項、沒有說明、沒有本機資料，V0 的照片預覽與照片匯出仍必須可正常使用。
- 每次修改後必須執行 V0 regression checklist。
- 表格輸出邏輯、照片處理邏輯、UI 邏輯必須分離。
- 資料儲存邏輯必須封裝在 repository 或 storage 層。
- UI 不得直接操作 IndexedDB 或 localStorage。
- repository 或 storage 層必須保留未來替換成後端資料庫的空間。
- 每次只實作一個小功能。
- 不要一次重寫整個專案。
- 不要新增不必要套件。
- 不要新增重複功能或重複 helper function。
- 修改前先搜尋是否已有相同功能。
- 每個模組必須有清楚責任。
- 優先使用瀏覽器原生能力。
- 優先採用可直接搬移到其他電腦執行的結構。

## Forbidden Actions

- 不要為 V1 刪除或改壞 V0 原本的照片處理與匯出流程。
- 不要讓工項、說明、IndexedDB 或 localStorage 成為 V0 匯出的必要條件。
- 不要在 V1 加入 AI、RAG、自動學習、帳號權限或後端服務。
- 不要為未確認需求建立大型框架。
- 不要把 UI、照片處理、匯出表格邏輯混在同一段不可拆分流程中。
- 不要引入大型依賴來處理瀏覽器原生可完成的功能。
- 不要刪除或覆蓋使用者既有檔案，除非使用者明確要求。
- 不要自行變更版本範圍。
- 不要在需求不明時假設完整產品功能。

## Testing and Acceptance

每次修改後必須通過：

- `docs/V0_REGRESSION_CHECKLIST.md`
- `docs/V1_CHECKLIST.md`，只在 V1 功能實作後適用

不得宣稱 V1 完成，除非 V0 regression checklist 與 V1 checklist 都通過。

## Token Saving Rules

- 不要把整個專案檔案都讀入 context。
- 優先讀取與本任務直接相關的檔案。
- 修改前先說明會動到哪些檔案。
- 大型功能必須拆成多個 task。
- 優先小範圍搜尋，不做無目的全專案掃描。
- 不重複讀取已經讀過且未變更的檔案。
- 回覆保持簡短，聚焦修改內容、測試方式與範圍確認。

## Done Definition

每個任務完成時必須說明：

- 修改了哪些檔案。
- 如何測試或驗收。
- 是否新增依賴。
- 是否有重複邏輯。
- 是否符合目前版本範圍。
- 是否保留前一版驗收條件。
